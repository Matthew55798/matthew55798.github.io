# 集合

Java 集合，也叫作容器，主要是由两大接口派生而来：一个是 `Collection`接口，主要用于存放单一元素；另一个是 `Map` 接口，主要用于存放键值对。对于`Collection` 接口，下面又有三个主要的子接口：`List`、`Set` 、 `Queue`。

# 数据结构

### List

- `ArrayList`：`Object[]` 数组。
- `Vector`：`Object[]` 数组。
- `LinkedList`：双向链表(JDK1.6 之前为循环链表，JDK1.7 取消了循环)。

### **Set**

- `HashSet`(无序，唯一): 基于 `HashMap` 实现的，底层采用 `HashMap` 来保存元素。
- `LinkedHashSet`: `LinkedHashSet` 是 `HashSet` 的子类，并且其内部是通过 `LinkedHashMap` 来实现的。
- `TreeSet`(有序，唯一): 红黑树(自平衡的排序二叉树)。

### Queue

- `PriorityQueue`: `Object[]` 数组来实现小顶堆。
- `DelayQueue`:`PriorityQueue`。
- `ArrayDeque`: 可扩容动态双向数组。

### Map

- `HashMap`：JDK1.8 之前 `HashMap` 由数组+链表组成的，数组是 `HashMap` 的主体，链表则是主要为了解决哈希冲突而存在的（“拉链法”解决冲突）。JDK1.8 以后在解决哈希冲突时有了较大的变化，当链表长度大于阈值（默认为 8）（将链表转换成红黑树前会判断，如果当前数组的长度小于 64，那么会选择先进行数组扩容，而不是转换为红黑树）时，将链表转化为红黑树，以减少搜索时间。详细可以查看：[HashMap 源码分析](https://www.notion.so/java/collection/hashmap-source-code.html)。
- `LinkedHashMap`：`LinkedHashMap` 继承自 `HashMap`，所以它的底层仍然是基于拉链式散列结构即由数组和链表或红黑树组成。另外，`LinkedHashMap` 在上面结构的基础上，增加了一条双向链表，使得上面的结构可以保持键值对的插入顺序。同时通过对链表进行相应的操作，实现了访问顺序相关逻辑。详细可以查看：[LinkedHashMap 源码分析](https://www.notion.so/java/collection/linkedhashmap-source-code.html)
- `Hashtable`：数组+链表组成的，数组是 `Hashtable` 的主体，链表则是主要为了解决哈希冲突而存在的。
- `TreeMap`：红黑树（自平衡的排序二叉树）。

重点：

Map

扩容：链表长度8 并且 数组长度64以上时候  转成红黑树

缩容：树节点小于6的时候恢复链表

平衡性： 红黑树是一种自平衡二叉搜索树，它通过特定的颜色标记和旋转操作保证了树的高度大致平衡，从而确保了最坏情况下的查找、插入和删除操作的时间复杂度为 O(log n)。
插入和删除性能： 相比于 AVL 树，红黑树在插入和删除操作时需要进行较少的旋转调整，因此在频繁更新的数据集上表现更好。AVL 树虽然能提供更严格的平衡，但维护这种平衡的成本较高。

内存占用： 红黑树只需要额外的一位颜色信息（红色或黑色）来维护平衡，相比一些其他复杂的平衡树结构，它的内存开销较小。

实现简单性： 红黑树的算法虽然复杂，但是已经被广泛研究并优化过，而且有标准的实现方法。这使得开发者可以更容易地理解和维护代码。

实际应用效果： 在实践中，红黑树能够很好地满足大多数场景下的性能需求。对于大多数数据集而言，红黑树提供的平衡性和性能已经足够优秀，不需要引入更复杂的结构。

综上所述，红黑树在平衡性、性能、内存效率以及实现复杂度之间找到了一个很好的平衡点，因此被选作 HashMap 中解决哈希冲突的一种有效手段。
当然，这并不意味着红黑树是所有情况下最优的选择；在某些特定的应用场景下，可能会有更适合的树结构或其他数据结构。但在 HashMap 的上下文中，红黑树是一个非常合适的选择。

红黑树：每个节点只需要一个额外的位来存储颜色信息，这使得红黑树在内存使用上非常高效。
AVL树：每个节点需要存储一个平衡因子，通常是一个整数，这会增加额外的内存开销。
B-树 和 B+树：每个节点需要存储多个键和指针，这会导致更高的内存开销。
Splay树 和 Treap：虽然不需要显式的平衡信息，但它们的操作可能涉及更多的旋转，这可能间接增加内存开销
