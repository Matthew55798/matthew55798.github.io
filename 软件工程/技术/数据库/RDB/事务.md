# 事务

# 事务的ACID特性

- **原子性**（`Atomicity`）：事务是最小的执行单位，不允许分割。事务的原子性确保动作要么全部完成，要么完全不起作用；
- **一致性**（`Consistency`）：执行事务前后，数据保持一致，例如转账业务中，无论事务是否成功，转账者和收款人的总额应该是不变的；
- **隔离性**（`Isolation`）：并发访问数据库时，一个用户的事务不被其他事务所干扰，各并发事务之间数据库是独立的；
- **持久性**（`Durability`）：一个事务被提交之后。它对数据库中数据的改变是持久的，即使数据库发生故障也不应该对其有任何影响。

原子性，隔离性和持久性是数据库的属性，而一致性（在 ACID 意义上）是应用程序的属性。**应用可能依赖数据库的原子性和隔离属性来实现一致性**，但这并不仅取决于数据库。因此，字母 C 不属于 ACID 。

# 事务隔离级别

| 隔离级别 | 英文名 | 特点 | 常见用途 |
| --- | --- | --- | --- |
| 1️⃣ 读未提交 | READ UNCOMMITTED | 允许读取未提交的数据（脏读） | 几乎不用，风险高 |
| 2️⃣ 读已提交 | READ COMMITTED | 允许读取并发事务已经提交的数据 | Oracle 默认级别 |
| 3️⃣ 可重复读 | REPEATABLE READ | 对同一字段的多次读取结果都是一致的，除非数据是被本身事务自己所修改 | MySQL 默认级别 |
| 4️⃣ 可串行化 | SERIALIZABLE | 所有事务串行执行（无并发） | 严格财务 |
- Oracle Database offers the read committed (default) and serializable isolation levels. Also, the database offers a read-only mode.
- Oracle 数据库默认提供 REPEATABLE READ 和 SERIALIZABLE，还有只读模式。

# 多事务并行的异常现象

| 异常现象 | 描述 | 示例 |
| --- | --- | --- |
| **1. 脏读 (Dirty Read)** | 事务 A 读到了事务 B 尚未提交的修改 | B 修改数据后未提交，A 读到这个修改；B 回滚后，A 读到的其实是“脏”数据。 |
| **2. 不可重复读 (Non-Repeatable Read)** | 同一个事务中两次读到的数据不一致 | A 第一次读数据，B 修改并提交后，A 再次读取同一行数据，结果不同。 |
| **3. 幻读 (Phantom Read)** | 同一个事务中，两次读取的行数不同 | A 读取满足条件的多行，B 插入或删除了部分行，A 再次读取时行数发生变化。 |
| **4. 丢失更新 (Lost Update)** | 两个事务同时修改同一条记录，后提交的覆盖了先提交的结果 | A、B 同时读取一行，分别修改并提交，A 的修改被 B 覆盖。 |

当隔离级别设置为Read uncommitted时，事务B读取了事务A尚未提交的数据，就可能出现脏读。

当隔离级别设置为Read committed时，事务B读不到事务A尚未提交的数据，避免了脏读，但是事务B读取了事务A已经提交的数据，可能会造成不可重复读。

当隔离级别设置为Repeatable read时，对**同一字段**的多次读取结果都是一致的，除非数据是被本身事务自己所修改，可以阻止脏读和不可重复读，但幻读仍有可能发生。MySQL InnoDB 存储引擎的默认隔离级别正是 REPEATABLE READ。并且，InnoDB 在此级别下通过 MVCC（多版本并发控制） 和 Next-Key Locks（间隙锁+行锁） 机制，在很大程度上解决了幻读问题。

| 隔离级别 / 异常类型 | 脏读 (Dirty Read) | 不可重复读 (Non-Repeatable Read) | 幻读 (Phantom Read) | 丢失更新 (Lost Update) |
| --- | --- | --- | --- | --- |
| **READ UNCOMMITTED** （读未提交） | ✅ 可能出现 | ✅ 可能出现 | ✅ 可能出现 | ✅ 可能出现 |
| **READ COMMITTED** （读已提交） | ❌ 阻止 | ✅ 可能出现 | ✅ 可能出现 | ✅ 可能出现 |
| **REPEATABLE READ** （可重复读） | ❌ 阻止 | ❌ 阻止 | ✅ **部分防止**出现 | ✅ 可能出现 |
| **SERIALIZABLE** （可串行化） | ❌ 阻止 | ❌ 阻止 | ❌ 阻止 | ❌ 阻止 |

# 多事务并行的安全措施

## 脏读

Oracle默认隔离级别 **READ COMMITTED** 读已提交，避免了脏读。

MySQL默认隔离级别 **REPEATABLE READ**  可重复读，避免了脏读。

## 不可重复读

MySQL默认隔离级别 **REPEATABLE READ**  可重复读，避免了不可重复读。

## 丢失更新

这是我们真正需要花时间解决的问题

业务场景：

1. 库存加减，账户转账
2. 计数投票点赞
3. 审批

解决方案：

1. **悲观锁**：`SELECT ... FOR UPDATE`在读取阶段加锁
2. **CAS+版本号/乐观锁**：表中增加version字段，使用版本号检查
    
    ```java
    - 查询
    SELECT id, value, version FROM product WHERE id = 1;
    - 更新
    UPDATE product
    SET value = ?, version = version + 1
    WHERE id = ? AND version = ?;
    ```
    
3. **原子操作**：在单个UPDATE语句中完成计算
4. **接口幂等：**含义是多次调用保证结果相同（不是说返回的结果，是业务数据最终状态），方案是设计业务状态机，使用业务状态字段控制流程，更新前检查状态是否匹配并加锁（行锁，分布式锁等），缺点是状态机流转路径设计缺陷会引发业务问题，或者代码写的不好会引起死锁。
5. **设置`SERIALIZABLE`隔离级别**
